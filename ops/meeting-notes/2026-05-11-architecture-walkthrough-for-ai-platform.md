# Architecture Walkthrough for AI Platform — 2026-05-11

## Summary
Deep dive into Betterworks' current AI architecture, led by Jason (JZ) with Nellie and Lalit. Jason presented the three-tier architecture — UI layer, business logic (LM Engine + LM Proxy), and backend infrastructure including the private LLM. He detailed the request workflow through the LM Engine and LM Proxy to the private LLM, guardrails for content safety, and the use of TensorRT for latency improvement. The discussion covered data infrastructure (AWS Aurora data warehouse, S3 data lake, Vector DB) and the preprocessing system using Airflow for scheduled tasks. Jason also explained the asynchronous skill normalization process and addressed latency and scheduled-processing questions.

## Key decisions / architecture facts
- Three-tier architecture: UI → Business Logic (LM Engine + LM Proxy) → Backend (Private LLM + Vector DB + Data Warehouse).
- Private LLM strategy: production moved from OpenAI to Meta Llama 3.1 8B hosted on AWS EC2 (GPU) via EKS; next migration target is Gemma 4. GuardRail uses the same model as generation.
- Guardrail mechanism: LM Proxy checks input and output by sending the prompt to the LLM to validate against ~20 violation categories. Output checks are skipped for some features (e.g., feedback/performance summary) because input was already checked.
- Performance: migrating inference from Python-based to TensorRT (NVIDIA) reduced writing-assistant latency from 7–10s to 3–5s and summarization from <1 min to <20s. TensorRT requires model-specific builds and does not yet support Gemma 4, so initial Gemma deployment will have longer latency until TensorRT support arrives.
- Data flow: Production DB → S3 Data Lake → Aurora Data Warehouse → LM Engine. Talent profile DB being separated for skill-related data.
- Retry: 3 retries on LLM call failure.
- Vector DB migrating from Chroma to pgvector (Saga working on it).
- Embedding provider: OpenAI today for skills normalization; future Gemma embedding to remove third-party vendors ("pure private").

## Discussion notes
- Core request flow: User → API Gateway → Legacy Core (monolith) → LM Engine → LM Proxy → Private LLM. LM Engine composes prompts (system prompt + user input + config); LM Proxy performs guardrail checks.
- Synchronous vs asynchronous patterns:
  - Real-time (sync): writing assistants execute immediately with user waiting.
  - Asynchronous: skills discovery posts a request and returns immediately; UI polls every 3 seconds until results are ready in DB; processing continues even if the user closes the window.
  - Pre-processing (scheduled): feedback and performance summaries run via AWS MWAA (Managed Airflow) weekly on weekends, iterating all users to pre-generate content stored in DB — eliminating runtime generation delay.
- Skills normalization workflow: Haven UI triggers skills discovery → Talent API → LM Engine retrieves conversation/feedback from CFG services → composes prompt → Private LLM generates raw skills → normalization service calls OpenAI embedding (future: Gemma) → clusters skills → stores in DB → UI polls for completion.
- AI features supported: feedback summary, goal writing assistant, conversation writing assistant, skills normalization.
- Writing-assistant sequence: client → API gateway → legacy core → LM Engine → private LLM or OpenAI depending on config, with guardrail input checks.
