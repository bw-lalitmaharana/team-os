# Webhooks — 2026-07-07

## Summary
Danish walked the team through his implementation of the inbound webhook gateway service, a platform component (not transcripts-only) that handles third-party integration events. The service uses an auth-proxy layer to manage per-vendor-org credentials, issuing unique webhook URLs for individually-provisioned vendors (Docebo, LinkedIn Learning) but requiring manual CLI provisioning for Zoom's shared-webhook/marketplace-app model. Events are enriched, deduplicated via Redis, and published to internal Kafka topics without schema validation (deferred to protect latency). Okan pushed for aligning the event schema to Cloud Events standards and flagged that Zoom's shared-webhook pattern may warrant its own JIRA ticket to determine if it's industry-standard or a one-off. The group agreed to position the gateway as a reusable platform component.

## Key decisions
- Webhook gateway positioned as a shared platform component, not a transcripts-only build.
- Zoom uses a shared/marketplace-app webhook model requiring manual CLI provisioning; Docebo/LinkedIn Learning get unique per-org webhooks via auth-proxy.
- No schema validation on published Kafka events for now — deferred to keep latency low; failure events will still be logged.
- Event schema field names to be aligned with the Cloud Events standard.
- JB (App Platform) and InfoSec review both required before Danish's PR merges.
